# 프로젝트 기본 규칙

이 프로젝트는 비개발자 강의 챌린지에서 만들어집니다. 모든 결정은 아래 규칙을 따릅니다.

## 기술 스택 (고정)
- Next.js (App Router) + TypeScript + Tailwind CSS
- Supabase (데이터베이스)
- Vercel (배포)
- Gemini API (AI 기능, .env.local 의 GEMINI_API_KEY 사용)
- 위 스택 외 다른 프레임워크·런타임·DB 추가 금지
- 라이브러리 추가는 최소화. 꼭 필요한 경우만, 한 줄 이유와 함께

## 경로 구조 (고정)
- / = 메인페이지 (서비스 소개, 정적 디자인)
- /app = 본 서비스 (데이터·AI 기능)
- 사용자가 다른 경로를 요청해도 위 둘이 충돌하지 않게 가이드

## 파일 규칙
- design.md / main-prd.md / service-prd.md = 사용자가 만든 사양 문서. 코드는 항상 이 문서와 일치 여부 확인
- 사용자가 명시적으로 요청하지 않은 파일·폴더는 건드리지 말 것
- .env.local 은 사용자가 직접 관리. 코드는 환경 변수만 읽고, 키 값을 코드에 박지 말 것

## 디자인 규칙
- design.md 항상 참고
- 폰트: Pretendard 유지 (다른 폰트 추가 금지)
- 기울임체 (italic) 사용 금지

## Phase 진행 규칙
- service-prd.md = Phase 1 (MVP) + Phase 2~5 (고도화)
- "Phase 1 만들어줘" → Phase 1만 빌드, Phase 2~5는 손대지 않음
- "Phase 2 얹어줘" → Phase 2만 추가, Phase 1과 충돌 없게
- Phase 1이 작동 안 하는 상태에서 Phase 2~5 추가하지 말 것 (사용자에게 알림)

## 응답 톤
- 모든 설명·주석은 한국어로
- "더 쉽게 알려줘" 요청 시 초등학생도 알 수 있는 비유로 다시 설명
- 기술 용어 사용 시 옆에 한 줄 풀이
- 큰 변경 전엔 진행 계획을 짧게 보여주고 OK 받기
- 작업 후엔 무엇이 바뀌었는지 1~3줄 요약

## 데이터 안전
- Supabase 테이블 스키마 변경 (컬럼 추가·삭제·이름 변경) 시 사용자에게 명시적 확인
- 기존 데이터 삭제·덮어쓰기는 사용자가 명시적으로 요청한 경우만
